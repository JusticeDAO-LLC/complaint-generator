How The Complaint Moves

A homepage that actually leads into the case workflow

This landing page is no longer pretending to be a login widget. It now mirrors the real case lifecycle and points each stage into the corresponding dashboard so a user can move through the complaint coherently.

1

Capture intake

Use the workspace to answer intake questions and preserve the complaint narrative in the DID-backed session.

2

Add evidence

Attach testimony and document support directly to claim elements instead of scattering proof across pages.

3

Review support posture

Open the review surface to see weak elements, missing support, and the recommended next evidence action.

4

Draft and refine

Generate the complaint, edit it in the drafting surfaces, and inspect optimization traces or UX audits when needed.

Shared Platform

One complaint engine, exposed everywhere

The complaint generator is not just a webpage. It is the same shared service contract surfaced through the Python package, CLI tools, MCP server tools, and the browser JavaScript SDK used by the dashboards.

Package exports

Import the complaint workspace

Python package exports expose session management, identity helpers, and closed-loop UI optimization workflows for automation.

CLI tools

Run the same workflow from the terminal

Use complaint CLI commands to inspect sessions, create identities, list tools, review UI artifacts, and run optimization loops.

MCP server + JS SDK

Use the same tools in the browser

The dashboards call the JSON-RPC MCP server through the JavaScript SDK, so the website shares the same tool paths as automation.
